# 레포 → Claude Design 동기화 — 빌드된 앱이 프로토타입과 달라진 지점

> 작성 2026-06-21 · 목적: 실제 `BBD-AES/mobile`(main)이 **API 연동·인증**을 붙이며 Claude Design 프로토타입과 달라진 부분을 Design 쪽에 반영(또는 결정)하기 위한 체크리스트.
> 전제: 화면 비주얼/스타일은 이미 프로토타입을 Compose로 포팅한 것. 아래는 **API/인증 연동 때문에 새로 생기거나 바뀐** 차이만.
> 동작 분기: 전 화면이 `USE_API` 플래그 뒤. **off(기본)=시드(프로토타입 그대로), on=실 API**.

## 1. 🆕 로그인 — OIDC 변형 추가 (Design에 없음 → 추가 필요)
- **프로토타입:** 사번+비밀번호 폼(데모 게이팅).
- **빌드된 앱(API 모드):** **Keycloak 로그인** — 로고 + "Keycloak으로 로그인" 버튼 → 브라우저(Custom Tab) OIDC. 사번/비번 입력 없음.
- **Design 할 일:** OIDC 로그인 화면 1장 추가(버튼형). 사번/비번 폼은 "데모 전용"으로 표기.

## 2. 🔄 입고(receive) — 부품 스캔 → **QR/SO 수령**으로 변경 (스캔 플로우 재설계)
- **프로토타입:** 부품 바코드 스캔 → 수량 입력 → "입고 등록"(부품 단위).
- **백엔드 현실:** 입고 = `PATCH /sales-orders/{soNumber}/receive` — **주문(발주) 단위** 상태 전환(IN_FULFILLMENT→RECEIVED). 부품·수량 단위가 아님.
- **빌드된 앱(API 모드):** 바코드/QR 스캔(또는 SO번호 직접 입력) → **그 발주를 입고 확정**.
- **Design 할 일:** 입고 스캔 화면을 **"발주 스캔→확정"**(주문 단위)로. 부품 스캔→수량 모델은 백엔드와 안 맞으니 폐기/데모표기. **출고 스캔은 백엔드 엔드포인트 없음** → 데모/보류 유지.

## 3. 🔄 목록 카드 — API 모드는 **주문 단위**로 단순화 (또는 백엔드 보강)
- **프로토타입:** 발주·작업이력 카드가 부품 단위(품목명·수량·썸네일).
- **백엔드 현실:** SO **목록 요약**에 라인(SKU·수량) 없음 → API 모드 카드는 **SO번호·상태·지점·합계·날짜**(주문 단위)로 렌더 중. 리치 카드는 시드에만 해당.
- **결정 필요(둘 중):**
  - (A) Design이 **API 모드용 주문 단위 카드**를 별도로 그림, 또는
  - (B) 백엔드가 요약에 `itemCount·totalQuantity` 추가(가벼움) → 리치 카드 유지 가능. (`mobile-backend-gaps.md` Gap 2)

## 4. 🆕 로딩 / 에러 / 빈 상태 (Design에 없음 → 명세 필요)
- 실 API라 화면마다 **로딩 스켈레톤 · 에러+다시시도 · 빈 상태**가 추가됨(현재 공통 컴포넌트로 구현).
- **Design 할 일:** 디자인 시스템에 3-상태(loading/error/empty) 명세 — 지금은 개발자 기본 스타일. (UX 감사 #8·#48과 연결)

## 5. 🔄 재고 — 백엔드 의존 + 카테고리 칩 데이터 도출
- 재고 화면은 inventory **목록 DTO 보강**(안전재고·카테고리·단위·가용재고) 반영 전까지 실데이터 미표시(`mobile-backend-gaps.md` Gap 1, 요청 발송됨).
- 카테고리 필터 칩은 **고정 목록이 아니라 실 재고의 카테고리에서 도출**(API 모드) — **Gap 1(목록 DTO에 카테고리 포함) 해결 후에야 구현 가능**. 그 전까지 Design의 고정 카테고리 칩은 시드 기준.
