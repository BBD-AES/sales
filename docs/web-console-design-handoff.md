# Handoff: BBD-AES ERP **웹 콘솔** — 디자인 변경 요청 (Claude Design)

> 출처: 리뷰 피드백 정리(2026-06-22). 대상 = `frontend-react` 웹 ERP 콘솔.
> 이 문서는 **디자인(UI·UX·IA·용어·역할별 화면) 요청사항만** 담습니다.
> 백엔드/인증·인가 아키텍처, 발표·시연 전략, 업무 시나리오 준비 등은 디자인 범위가 아니라 제외했습니다(맨 아래 "제외 항목" 참고).

---

## 1. 버그 / 동작 수정
- **상단 검색 동작 안 함** — 헤더 "SKU·주문번호·고객명 검색" 입력의 검색/결과 UI를 정의·연결.
- **사이드바 접힘(collapsed) 상태 디자인 깨짐** — 접었을 때 레이아웃/아이콘 정렬 수정.
- **판매주문 작성: HQ 매니저·스태프가 제출 불가** — 해당 역할도 작성·제출 가능하도록 폼/버튼 상태 정리.
- **PO(발주) 작성(드래프트) 상태에 수정 기능 없음** — 드래프트에서 라인/수량 편집 가능하도록 추가.

## 2. 정보구조(IA) — 페이지 통합
- **발주(PO) + 발주 요청 페이지 합치기.**
  - ⚠️ 정의 필요: 현재 "발주 요청" 페이지에 **정확히 무엇이 보여야 하는지** 불명확 → 합치면서 내용·역할 명세.
- **재고 요약 + 재고 현황 → 한 페이지로** (탭/섹션으로 통합). ※ 아래 §4 컴포넌트 변경과 함께.
- **생산: "생산 요청" 제거 → "생산 작업지시" 하나로 통합** (단일 화면).

## 3. 폼 / 플로우
- **판매주문 → PO 작성 시: 들어온 주문(수주/CO)만 드롭다운에 노출**하고 그 주문을 기준으로 발주하도록. (자유 입력 X, 들어온 주문 선택 기반.)
- **보충 요청 등록에 CO(고객주문) 선택 버튼 추가** — 어떤 고객주문에 대한 보충인지 선택.
- **작업지시: SO 번호 필수 여부 재검토** — 지금 필수인데 선택으로 둘지 결정 필요(정의 필요).

## 4. 화면별 컴포넌트 (재고)
- **재고 요약**: "품목이 몇 종류인지(종류 수)" 지표는 비중요 →
  - **종류를 볼 수 있는 드롭다운**, 또는 **종류별 수량** 형태로 재구성.
- **재고 현황**: 컬럼 순서 변경 — **SKU 대신 창고(warehouse)를 맨 앞으로**.

## 5. 용어 정리 (라벨 일원화)
- **"판매 주문" → "재고 이동 요청"** 으로 변경.
- **"보충 요청 등록" 의 실제 의미 = purchase request** → 명칭 정합.
  - ⚠️ 정의 필요: 위 두 항목(재고 이동 요청 / purchase request)이 **같은 엔티티인지 다른지** 정리해 용어 체계를 한 번에 일원화.

## 6. 역할 기반 화면 / 대시보드 / 프로세스 가시성
- **역할별 사이드바 통제** — 로그인 역할과 무관한 업무 메뉴는 숨김.
  - (참고: 현재 빌드는 BRANCH_STAFF(정비사)로 로그인해도 전 메뉴(대시보드·품목·재고·영업·구매·생산)가 그대로 노출됨 → 역할 게이팅 필요.)
- **역할별 대시보드 차별화** — 진행 상황 중심으로 다르게. **매니저·어드민 = 풀(full) 대시보드.**
- **어드민** — 로그인 시 **바로 "사용자 관리"로** 진입. (어드민 = 전 권한 전제.)
- **프로세스 진행 가시성 강화** — 주문/입고/생산 등 진행 중인 단계가 어디서나 잘 보이도록.
- **단계 완료 알림** — 각 단계가 완료되면 완료 알림을 표시(이벤트 기반 도착 알림 UX).

## (참고) 이 핸드오프에서 **제외한** 항목 — 디자인 범위 아님
- **인증/인가 아키텍처**: Keycloak 유저 분리/통합계정 연동, 게이트웨이의 리소스서버 위치, 세션(JSESSIONID)·Redis 세션 공유, 토큰을 리다이렉트로 전달, 게이트웨이의 MSA 적합성 등 → **백엔드 설계**.
- **발표/시연 전략**: 이벤트드리븐 강조 시연, 다이어그램 보강, 프로세스 설명 순서, 모바일 입고 시연만 보여주기, 업무 시나리오·재고정산 "말로 때우기", MM/자재관리 포지셔닝 어필 등 → **발표 준비**.
- **아키텍처 판단/표현**: 트랜잭션-이벤트 결합 회의론, 단방향에 이벤트 부적절, 비동기 위험, "예약=선점" 표현 등 → **설계/발표 화법**.
